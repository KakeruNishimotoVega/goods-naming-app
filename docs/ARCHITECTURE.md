# アーキテクチャドキュメント

## システム全体像

```
┌─────────────────────────────────────────────────────────────┐
│                    Google Apps Script                        │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                  フロントエンド                         │  │
│  │  ・index.html (Tailwind CSS)                          │  │
│  │  ・ユーザーインタラクション                              │  │
│  │  ・google.script.run でバックエンド呼び出し             │  │
│  └───────────────────────────────────────────────────────┘  │
│                            ↕                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                バックエンドAPI                          │  │
│  │  ・src/api/* (TypeScript)                             │  │
│  │  ・ビジネスロジック                                      │  │
│  │  ・UrlFetchApp でSupabase REST API呼び出し             │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                            ↕ HTTPS
┌─────────────────────────────────────────────────────────────┐
│                    Supabase (PostgreSQL)                     │
│  ・Categories, Types, Keywords, Regulations, etc.           │
│  ・REST API / PostgREST                                     │
└─────────────────────────────────────────────────────────────┘
```

## レイヤー構造

### 1. プレゼンテーション層（フロントエンド）
**ファイル**: `src/index.html`

- **責務**: ユーザーインターフェース、イベントハンドリング
- **技術**: HTML, Tailwind CSS, Vanilla JavaScript
- **通信**: `google.script.run` でバックエンド関数を呼び出し
- **特徴**: SPAライクな画面遷移（テンプレート切り替え）

### 2. アプリケーション層（バックエンドAPI）
**ディレクトリ**: `src/api/`

各機能領域ごとにファイルを分割：

| ファイル | 責務 |
|---------|------|
| `categories.ts` | カテゴリのCRUD操作 |
| `types.ts` | 入力項目（Type）の管理 |
| `keywords.ts` | 選択肢（Keyword）の管理 |
| `regulations.ts` | 命名ルール（Regulation）の管理 |
| `ngwords.ts` | NGワードの管理 |
| `naming.ts` | 商品名・ページ名の生成ロジック |

### 3. インフラ層（共通ライブラリ）
**ディレクトリ**: `src/lib/`

| ファイル | 責務 |
|---------|------|
| `supabase.ts` | Supabase REST API呼び出しのヘルパー |
| `utils.ts` | 汎用ユーティリティ関数 |

### 4. データ層（Supabase）
**管理**: Supabase Dashboard + MCP Tools

- PostgreSQLデータベース
- REST API（PostgREST）経由でアクセス
- Row Level Security（RLS）で権限管理

## モジュール分割の原則

### ファイル分割のルール
1. **1ファイル = 1機能領域**: カテゴリ、Type、Keywordなど
2. **最大500行を目安**: ファイルが大きくなりすぎたら分割検討
3. **依存関係は一方向**: 上位レイヤーから下位レイヤーへ

### 命名規則
- **ファイル名**: kebab-case（`categories.ts`, `ng-words.ts`）
- **関数名**: camelCase（`getCategories`, `addType`）
- **型名**: PascalCase（`CategorySchema`, `TypeData`）
- **定数**: UPPER_SNAKE_CASE（`MAX_NAME_LENGTH`）

## データフロー

### 典型的なリクエストフロー

```
1. ユーザーがフロントエンドで操作
   ↓
2. JavaScriptイベントハンドラーが発火
   ↓
3. google.script.run.関数名() でバックエンド呼び出し
   ↓
4. バックエンド関数（src/api/*）がリクエスト処理
   ↓
5. src/lib/supabase.ts 経由で Supabase REST API 呼び出し
   ↓
6. Supabaseがデータを返却
   ↓
7. バックエンドが結果を整形してフロントエンドに返却
   ↓
8. フロントエンドが結果を画面に反映
```

### エラーハンドリングフロー

```
エラー発生
   ↓
try-catchでキャッチ
   ↓
Logger.log() にデバッグ情報を出力
   ↓
ユーザーフレンドリーなエラーメッセージを返却
   { error: "カテゴリの取得に失敗しました", details: ... }
   ↓
フロントエンドでエラー表示（モーダルまたはトースト）
```

## GAS特有の制約と対応策

### 制約1: Node.jsモジュールが使えない
- **問題**: `@supabase/supabase-js` が動作しない
- **対応**: `UrlFetchApp` で Supabase REST API を直接呼び出し
- **実装**: `src/lib/supabase.ts` にラッパー関数を作成

### 制約2: グローバルスコープへのエクスポートが必要
- **問題**: フロントエンドから呼び出す関数はグローバルに公開が必要
- **対応**: `src/index.ts` で `(global as any).関数名 = 関数名;`

### 制約3: 環境変数の扱い
- **問題**: `.env` ファイルが使えない
- **対応**: `PropertiesService.getScriptProperties()` で環境変数を取得
- **設定方法**: GASスクリプトプロパティに `SUPABASE_URL`, `SUPABASE_KEY` を登録

### 制約4: ビルドプロセスが必要
- **問題**: TypeScriptやモジュールをGASで実行するには事前バンドルが必要
- **対応**: `build.js` + `esbuild` でバンドル、`clasp` でデプロイ

## セキュリティ考慮事項

### 1. 環境変数の管理
- Supabase URLとAPIキーはスクリプトプロパティで管理
- フロントエンドには**公開可能なキー**のみ渡す（anon key）
- サービスロールキーは絶対にフロントエンドに渡さない

### 2. Supabase RLS（Row Level Security）
- データベース側で権限制御を実装
- GAS経由でアクセスする場合も RLS が適用される

### 3. 入力値のバリデーション
- フロントエンドだけでなくバックエンドでも検証
- SQLインジェクション対策（Supabase REST APIが自動対応）

## パフォーマンス最適化

### 1. データ取得の最適化
- 必要なカラムだけを取得（`select=id,name`）
- ページネーション対応（将来的に）
- キャッシュ戦略の検討（GAS CacheService）

### 2. フロントエンドの最適化
- 不要なDOM操作を避ける
- イベントリスナーのデバウンス/スロットル
- 画像の遅延読み込み

### 3. ビルドサイズの削減
- 使用していないコードの削除（tree-shaking）
- CSS/JSの最小化（esbuild）

## 今後の拡張性

### スケーラビリティ
- ユーザー数が増えた場合 → Supabaseのプラン変更で対応
- データ量が増えた場合 → インデックス最適化、パーティショニング

### 機能追加の方針
1. 新しい機能領域 → `src/api/` に新ファイル追加
2. 既存機能の拡張 → 該当ファイル内に関数追加
3. 共通処理 → `src/lib/` に抽出

### 他システムとの連携
- REST API化を検討（GASのWeb Appとして公開）
- Webhookによる外部システム連携
- Zapier/Makeによる自動化

## AI保守運用のための設計

### ファイルの見つけやすさ
- 機能名でファイルが推測できる命名
- 階層は浅く（最大3階層）

### コードの理解しやすさ
- 関数は短く（最大50行を目安）
- 複雑なロジックには必ずコメント
- 型定義を徹底（any型の禁止）

### 変更の影響範囲の限定
- 疎結合な設計（各モジュールの独立性）
- 共通処理の集約（DRY原則）
- インターフェースの安定性
